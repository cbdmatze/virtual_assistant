Introduction

Welcome to the Bullseye API documentation! This API provides a range of functionalities, now organized into modules for better structure and maintainability. Key features include:

  • Chat with Multiple LLMs: Interact with various Large Language Models (LLMs) from providers like OpenAI, Anthropic, Groq, and Google Gemini.
  • Google Search Integration: Leverage Google Search for information retrieval within chat conversations, powered by LangChain and direct search methods.
  • Hugging Face Model: Utilize a local Hugging Face model for text generation.
  • Image Generation: Generate images based on text prompts using a RapidAPI service.
  • Optical Character Recognition (OCR): Extract text from uploaded images using Tesseract OCR.
  • User Authentication: Secure API access using OAuth2 Password Bearer token authentication.
  • Conversation History: Manage and retrieve conversation history stored in a MySQL database.

This documentation will guide you through each endpoint, detailing request methods, parameters, request/response bodies, authentication requirements, and error handling. Please note that due to recent backend modularization, endpoint paths and internal logic might have been adjusted. This documentation aims to reflect the most current API structure, but please verify endpoint details against your updated backend codebase if you encounter discrepancies.

Base URL

The base URL for all API endpoints is: http://localhost:8000 (assuming you are running the application locally with the default configuration). Please verify this base URL if your setup has changed with the backend modularization.

Authentication

This API uses OAuth2 Password Bearer authentication to secure access to most endpoints.

Obtaining an Access Token (/token)

POST /token

Content-Type: application/x-www-form-urlencoded

To authenticate and obtain an access token, you need to send a POST request to /token with your username and password.

Request Body Parameters (Form Data):
Parameter Type Required Description
username string Yes Your registered username.
password string Yes Your account password.
Example Request (using curl):
curl -X POST \
  -d "username=your_username&password=your_password" \
  http://localhost:8000/token
Successful Response (Status Code: 200 OK):
{
  "access_token": "1",  // User ID as access token (Bearer token)
  "token_type": "bearer"
}
Error Responses:
  • Status Code: 400 Bad Request: Incorrect username or password. 
    {"detail": "Incorrect username or password"}

Registering a User (/register)

POST /register

Content-Type: application/json

To register a new user, send a POST request to /register with the desired username and password.

Request Body (JSON):
{
  "username": "new_username",
  "password": "new_password"
}
Example Request (using curl):
curl -X POST \
  -H "Content-Type: application/json" \
  -d '{"username": "new_user", "password": "secure_password"}' \
  http://localhost:8000/register
Successful Response (Status Code: 200 OK):
{
  "message": "Registration successful"
}
Error Responses:
  • Status Code: 400 Bad Request: Username already exists. 
    {"detail": "Username already exists"}

Using the Access Token

Once you obtain an access_token, you must include it in the Authorization header as a Bearer token for all subsequent requests to protected endpoints.

Example Header:
Authorization: Bearer 1

Endpoints

1. Chat with LLMs (/chat)

POST /chat

Authentication: Required (OAuth2 Bearer Token)

Content-Type: application/json

This endpoint allows you to send a prompt to various LLMs and receive a text-based response. Verify if the endpoint path /chat is still correct after backend changes.

Request Body (JSON):
{
  "prompt": "Write a short poem about the ocean.",
  "model": "gpt-3.5-turbo",
  "temperature": 0.7,
  "api_provider": "openai"
}
Valid api_provider values:
  • "openai": OpenAI models (e.g., gpt-3.5-turbo, gpt-4).
  • "anthropic": Anthropic Claude models (e.g., claude-3-5-sonnet-20241022).
  • "groq": Groq models (e.g., llama-3.2-90b-vision-preview).
  • "google": Google Gemini models (e.g., gemini-1.5-pro).
  • "huggingface": Hugging Face gpt2 model (model name in request is ignored, always uses gpt2).
  • "langchain": LangChain Google Search agent (model name in request is ignored, always uses gpt-3.5-turbo-instruct and Google Search). Frontend should send "google-search" as model name for clarity.
Example Request (using curl):
curl -X POST \
  -H "Authorization: Bearer 1" \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Explain quantum physics simply.", "model": "gpt-3.5-turbo", "temperature": 0.6, "api_provider": "openai"}' \
  http://localhost:8000/chat
Successful Response (Status Code: 200 OK):
{
  "response": "Quantum physics is the study of the incredibly tiny world of atoms and particles. It's weird because things aren't always in one place or state, but rather exist in probabilities until measured. Imagine a coin spinning in the air – it's neither heads nor tails until it lands. That's a bit like quantum particles!"
}

6. Upload Image for OCR (/upload-image)

POST /upload-image

Authentication: Required (OAuth2 Bearer Token)

Content-Type: multipart/form-data

Uploads an image file to perform Optical Character Recognition (OCR) and extract text from it. Verify if the endpoint path /upload-image is still correct after backend changes.

Request Body (Form Data):
Parameter Type Required Description
file file Yes The image file to upload (e.g., PNG, JPG).
Example Request (using curl):
curl -X POST \
  -H "Authorization: Bearer 1" \
  -H "Content-Type: multipart/form-data" \
  -F "file=@/path/to/your/image.png" \
  http://localhost:8000/upload-image
Successful Response (Status Code: 200 OK):
{
  "message": "Image uploaded and text extracted successfully",
  "ocr_text": "This is the text extracted from the image using OCR."
}

Error Handling

The Bullseye API uses standard HTTP status codes to indicate the success or failure of API requests. In case of an error, the API will typically return a JSON response body with a detail key providing more information about the error.

Common Status Codes:

  • 200 OK: Request was successful.
  • 400 Bad Request: The request was malformed or invalid. Check the request parameters and body.
  • 401 Unauthorized: Authentication is required and has failed or has not yet been provided. Ensure you are sending a valid Bearer token in the Authorization header.
  • 404 Not Found: The requested resource could not be found. This might be more relevant if endpoint paths have changed due to modularization. Verify endpoint paths.
  • 500 Internal Server Error: An unexpected error occurred on the server. This could be due to issues with the API itself or with an upstream service. Check the server logs for more details.

For specific endpoints, refer to the 'Error Responses' sections for examples of error responses.

API Keys and Setup

To utilize the full functionality of the Bullseye API, you will need to set up API keys for the following services:

  • OpenAI API Key: Required for accessing OpenAI models like GPT-3.5 Turbo and GPT-4. Obtain your API key from the OpenAI platform.
  • Anthropic API Key: Required for accessing Anthropic Claude models. Obtain your API key from the Anthropic console.
  • Groq API Key: Required for accessing Groq models. Obtain your API key from the Groq console.
  • Google API Key & CSE ID: Required for Google Search integration and Google Gemini models.
    • Google API Key: Obtain an API key from the Google Cloud Console. Enable the "Custom Search API" and "Gemini API" for your project.
    • CSE ID (Custom Search Engine ID): Create a Custom Search Engine at Google Custom Search Engine and obtain the CSE ID.
  • RapidAPI Key: Required for image generation. Obtain your RapidAPI key after subscribing to the "ChatGPT Vision" API on RapidAPI.

Configuration in Backend (.env file or environment variables):

The backend application expects these API keys to be set as environment variables or within a .env file in the root directory of the project. While the provided backend code includes hardcoded API keys for demonstration purposes, it is highly recommended to use environment variables or a .env file for security in a production environment.

Example .env file structure:

OPENAI_API_KEY=your_openai_api_key
ANTHROPIC_API_KEY=your_anthropic_api_key
GROQ_API_KEY=your_groq_api_key
GOOGLE_API_KEY=your_google_api_key
GOOGLE_CSE_ID=your_google_cse_id
RAPIDAPI_KEY=your_rapidapi_key

Ensure you replace your_api_key and your_cse_id with your actual API keys and CSE ID.

Dependencies

The Bullseye API backend is built using Python and relies on several libraries. Ensure you have the following Python packages installed. You can install them using pip:

pip install fastapi uvicorn python-multipart pydantic mysql-connector-python Werkzeug Pillow openai anthropic groq google-generativeai pytesseract requests transformers langchain

Note: langchain and related libraries are optional and required only for LangChain Google Search agent functionality. If you do not intend to use LangChain, you can omit these dependencies. However, for full functionality, it is recommended to install them. Verify if the dependencies list is still accurate after backend modularization, especially if new modules introduced new dependencies.

Tesseract OCR: For the /upload-image endpoint (OCR functionality), you need to have Tesseract OCR engine installed on your system. Installation instructions vary by operating system. Please refer to the Tesseract OCR documentation for installation guides.

Running the Application

To run the Bullseye API application, follow these steps:

  1. Install Dependencies: Ensure you have installed all the necessary Python dependencies as listed in the Dependencies section.
  2. Set up API Keys: Configure your API keys as environment variables or in a .env file as described in the API Keys and Setup section.
  3. Database Setup: Make sure you have a MySQL database set up and configured as per the connection details in the backend code (host, user, password, database in the mysql.connector.connect function). Create a database named tasos (or change the name in the backend code) and ensure the user matthias (or your configured user) has the necessary permissions. The application will automatically attempt to add missing columns (temperature, model, image_data, api_provider) to the conversations table if they do not exist. You will need to create the users and conversations tables if they don't exist. Example SQL to create tables: 
    -- SQL to create users table
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(255) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL
);

-- SQL to create conversations table
CREATE TABLE conversations (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    conversation TEXT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    model VARCHAR(255),
    temperature FLOAT,
    image_data LONGTEXT,
    api_provider VARCHAR(50) DEFAULT 'openai',
    FOREIGN KEY (user_id) REFERENCES users(id)
);
  4. Run the FastAPI Application: Navigate to the directory containing the main.py file (or the main entry point of your modularized backend) in your terminal and run the command: 
    uvicorn main:app --reload --host 0.0.0.0 --port 8000
    • main: Refers to the main.py file. If your main application file or FastAPI instance name changed with modularization, update this command accordingly.
    • app: Refers to the FastAPI instance named app in main.py. If your FastAPI instance name changed, update this command accordingly.
    • --reload: Enables automatic reloading of the server upon code changes, useful for development.
    • --host 0.0.0.0: Binds the server to all network interfaces, making it accessible from other devices on the network.
    • --port 8000: Starts the server on port 8000 (default).
  5. Access the API: Once the application is running, you can access the API endpoints at http://localhost:8000. Verify this URL is still correct.

Testing Endpoints

You can test the API endpoints using various tools like curl, Postman, or Insomnia. Example curl commands are provided within each endpoint documentation. Here are general steps for testing:

  1. Obtain an Access Token: First, register a user using the /register endpoint and then obtain an access token using the /token endpoint with your registered username and password. Verify these authentication endpoints are still functional.
  2. Send Requests: Use curl or your preferred API testing tool to send requests to the API endpoints. Remember to include the Authorization: Bearer your_access_token header for protected endpoints, replacing your_access_token with the token you obtained. When testing, pay close attention to the endpoint paths as they might have changed.
  3. Verify Responses: Check the HTTP status codes and response bodies to ensure the API is functioning as expected. Compare the responses with the example successful and error responses provided in the documentation.
  4. Testing Specific Functionality:
    • Chat Endpoint: Test different prompts, models, and API providers using the /chat endpoint. Verify endpoint path and request/response structure.
    • Image Generation: Test image generation with various prompts and parameters using the /generate-image endpoint. Verify endpoint path and request/response structure.
    • Conversation History: Verify that conversations are being saved and retrieved correctly using the /conversations endpoint. Test deleting conversations using /conversations/{conversation_id} and /conversations (delete all). Verify endpoint paths and response structures.
    • OCR: Test the /upload-image endpoint by uploading image files and verifying the extracted text. Verify endpoint path and request/response structure.
    • Health Check & Test Endpoints: Use the /health, /test-google-search, and /test-langchain-search endpoints to monitor the API's health and test specific functionalities. Verify endpoint paths and response structures.
